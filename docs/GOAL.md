## TL;DR

**Clawvis is a self-hosted control center for AI agents.**

It solves a specific problem: after a few prompts, agents drift, context is lost, nothing is traceable. Clawvis enforces a frame — project, kanban, memory, logs — so agents stay autonomous *and* auditable.

**What makes it unique:**
- The kanban is the exact mirror of memory (PARA format) — not two separate tools
- Skills (OpenClaw crons) run tasks autonomously: innovation, implementation, morning briefing
- Everything is self-hosted, sovereign, traceable

**Primary audience:** solopreneur / solo dev who already uses LLMs but loses control without structure.

**The aha moment:** in 10 minutes you send a message → Clawvis structures, plans, and runs a full PoC. No chaos. Something concrete.

## Workflow actors

| Actor | Role |
|--------|------|
| **@Ldom** | The user — solo entrepreneur, steers projects via Telegram and the Clawvis Hub |
| **@Dombot** | The AI agent — OpenClaw instance configured by @Ldom, runs tasks via Clawvis skills |
| **Clawvis** | The framework — provides structure (projects, kanban, memory, logs) and skills @Dombot uses |

**Communication channels:**
- **Telegram** → direct @Ldom ↔ @Dombot (questions, confirmations, links)
- **Discord** → structured logging by topic channels (`#innovation`, `#projects`, `#logs`, `#ops`)

**Core rule:** Telegram = conversation. Discord = observability. If @Ldom does not need to act, they get no Telegram message.

## Main scenario: from idea to iterative PoC

### 1. Trigger — @Ldom has an idea

@Ldom sends a **voice message** to @Dombot on Telegram:

> *"I have a SaaS idea to automate freelance quotes with AI. Start a project."*

---

### 2. Initialization — @Dombot runs

@Dombot transcribes the voice and triggers the **`project-init`** skill:

1. Creates the project in Clawvis (slug, name, description from the voice)
2. Creates the GitHub repo (private repo linked to the project)
3. Creates the project page in the Brain (PARA: context, objective, resources, archive)
4. Generates the initial kanban (estimated tasks, `To Start` / `In Progress` / `Done` statuses)
5. Produces a first PoC (tech chosen by the agent from context)

**Telegram** → `"Project 'devis-ai' created ✅ — PoC available at lab.dombot.tech/apps/devis-ai/"`
**Discord `#projects`** → `New project: devis-ai — 8 tasks created — PoC generated — [Hub link]`

---

### 3. Exploration — @Ldom discovers the project

@Ldom opens the **Clawvis Hub**:

- **Project view** → kanban, estimates, overall status
- **Brain** → memory page: context, objective, decisions made by the agent
- **PoC** → prototype generated by @Dombot
- **Chat** → direct questions:
  > *"Why did you pick React and not Vue?"*
  > @Dombot explains using the memory page

@Ldom moves some tasks to **`To Start`** → makes them eligible for the next `kanban-implementer` cycle.

---

### 4. Autonomous iteration — @Dombot runs in the background

Every **4 hours**, the **`kanban-implementer`** skill runs (OpenClaw cron).

`kanban-implementer` orchestrates the full session (selection, status updates, PR, logs) then delegates unit execution to the **`implement`** skill:

1. Reads tasks in `To Start`
2. Picks the first priority task
3. Implements it using the Brain
4. Updates the PoC
5. Moves the task to `Done`

**Telegram** → 1 message per session (not per task): `"Session complete ✅ — 2 tasks done — PoC updated"`
**Discord `#logs`** → `[implement] devis-ai — task #3 done — 47 lines added — [GitHub diff]`

---

### 5. Validation — @Ldom validates and @Dombot ships

@Ldom reviews changes in the Hub, validates (or asks for fixes via chat).

Once validated, @Dombot:
- Pushes to GitHub (**`git-sync`** skill)
- Updates the Brain (**`knowledge-consolidator`** skill)
- Logs on **Discord `#projects`** (**`logger`** skill): `devis-ai — PoC v0.2 merged — 3/8 tasks complete`

Background tasks: **`brain-maintenance`** keeps Brain structure healthy, **`qmd`** handles structured memory writes.

## Additional scenarios

### 6. Morning briefing — strategic decision

**Trigger:** daily cron → **`morning-briefing`** skill

`morning-briefing` consolidates all active projects (`In Progress`, `Blocked`, `Done`), system metrics, tech watch if enabled.

**Telegram** → 1 single message:
> *"☀️ Brief 03/31 — devis-ai: 2 tasks done, 1 blocked; clawvis: no activity; system: OK. Reply to reorient."*

@Ldom replies: *"Pause devis-ai, focus on clawvis."*

@Dombot: moves `To Start` tasks from `devis-ai` to `Backlog`, raises `clawvis` priorities to `To Start`, updates the Brain, confirms on Telegram, logs on **Discord `#ops`**.

**What this scenario shows:** `morning-briefing` is the only mandatory daily touchpoint. @Ldom steers strategy from Telegram without opening the Hub.

---

### 7. Proactive innovation — draft project

**Trigger:** weekly cron → **`proactive-innovation`** skill

The skill detects a duplicated pattern across projects and spots an opportunity.

1. Publishes a structured brief on **Discord `#innovation`** (context, proposal, estimated effort, Brain link)
2. Creates a **`draft`** project in Clawvis (generated Brain page, empty kanban, hidden from main view)
3. **Telegram** → 1 decision message: `"💡 Opportunity: 'invoice-parser' — draft project created. [Hub link]"`

If @Ldom **accepts** → `active` status, kanban generated, eligible for `kanban-implementer`
If @Ldom **rejects** → draft archived, decision traced in the Brain

**What this scenario shows:** the agent creates value without being asked. `draft` mode keeps the Hub readable.

---

### 8. Weekly review — consolidation

**Trigger:** weekly cron → **`knowledge-consolidator`** skill

Consolidates completed tasks, velocity, Brain decisions, recurring blockages. Writes to global `WEEKLY.md` or `MEMORY.md`.

**Discord `#projects`** → per-project summary
**Telegram** → 1 minimal message: `"📋 Weekly review available → [Brain link]"`

**What this scenario shows:** Telegram carries the short signal. The Brain is the long-term source. Roadmap decisions happen in the Hub.

---

### 9. New project from existing work — reuse

**Trigger:** Telegram voice → **`project-init`** skill

Before creating, `project-init` reads the global Brain for reusable components and decisions:
- Creates the project with explicit links to parent projects
- Initial kanban excluding already-covered pieces
- Brain note: *"Reuse invoice-parser v0.3 — no need to reimplement"*

**Telegram** → `"Project 'client-tracker' created ✅ — 2 components reused from devis-ai. [link]"`
**Discord `#projects`** → reuse traceability

**What this scenario shows:** the more projects exist, the more the Brain makes `project-init` relevant. Reuse is explicit and traced.

## Components and their role in workflows

| Component | Role |
|-----------|------|
| **Kanban** | Control center — source of truth for what remains to be done |
| **Brain (Memory)** | Long-term memory — the agent feeds on it each iteration, avoids drift |
| **Skills (crons)** | Autonomous workflows triggered by event or schedule |
| **Chat** | Direct @Ldom ↔ @Dombot interaction in project context |
| **Logs** | Full traceability — every action visible in Hub and Discord |
| **Hub UI** | Visual entry point — projects, kanban, Brain, PoC in one place |
| **`draft` mode** | Opportunity backlog — visible for decisions, no noise in main flow |

## Clawvis skills overview

| Skill | Trigger | Role |
|-------|---------|------|
| `project-init` | Event (voice/message) | Full project (slug, repo, Brain, kanban, PoC) |
| `kanban-implementer` | 4h cron | Orchestrates session (selection, statuses, PR, logs) → delegates to `implement` |
| `implement` | Called by `kanban-implementer` | Unit execution of selected task |
| `morning-briefing` | Daily cron | Consolidated brief → Telegram + strategic decision |
| `proactive-innovation` | Weekly cron | Spots opportunities → draft project + Discord `#innovation` |
| `knowledge-consolidator` | Weekly cron + post-validation | Aggregates knowledge into `MEMORY.md` |
| `brain-maintenance` | Background | Maintains Brain structure and health on disk |
| `git-sync` | Post-validation | Syncs workspace and repo without committing secrets |
| `logger` | Post-action | Centralizes and structures logs to Discord |
| `qmd` | Memory writes | Reads/writes QMD memory files for the Brain |
| `skill-tester` | CI / manual | Tests skills and returns health report |
| `reverse-prompt` | Manual | Derives good prompts from observed behavior |

## Invariants — respect in every implementation

- **Everything is traced.** Every @Dombot action produces a visible log (Hub + Discord).
- **The Brain is the agent’s memory.** @Dombot reads the Brain before acting — never from zero.
- **The kanban is the source of truth.** Project state = kanban state. No actions outside the kanban.
- **@Ldom stays in control.** The agent proposes and executes; @Ldom validates before any definitive push.
- **Telegram = conversation, Discord = observability.**

## Cross-cutting constraints — signal vs noise

### Telegram: 1 message max per significant event

| Situation | Expected behavior |
|-----------|-------------------|
| Internal cron (implementation, Brain maintenance) | No Telegram — Discord only |
| Complete `kanban-implementer` session | 1 summary message (not 1 per task) |
| Project created | 1 confirmation with link |
| Weekly review | 1 minimal message with Brain link |
| Blockage or critical error | 1 alert with a precise question |

**Golden rule:** if @Ldom does not need to act, they get no Telegram message.

### Hub: multi-project readability

| Mechanism | Description |
|-----------|-------------|
| `draft` status | Unvalidated projects, hidden from main view |
| `backlog` status | Paused projects, visible but collapsed (no tasks) |
| Single project view | Per-project navigation — no aggregated global kanban |
| `In Progress` limit | `kanban-implementer` never promotes more than N tasks (default: 2, configurable) |
| Suggested archive | `morning-briefing` suggests archive after X days of inactivity |
