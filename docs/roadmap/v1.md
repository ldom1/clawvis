### 2B — OpenClaw + Clawvis on Hostinger VPS (prod, non-technical)

> Full guide: `docs/guides/deploy-hostinger.md`

**Hostinger tooling:**
- **Docker Manager** in hPanel: "Compose from URL" → paste raw GitHub `docker-compose.yml` URL, set env vars in UI, click Deploy. No terminal required.
- **GitHub Actions**: workflow `.github/workflows/deploy-hostinger.yml` (included) — auto redeploy on each push. Requires `HOSTINGER_API_KEY` + `HOSTINGER_VM_ID` as GitHub secrets and `HOSTINGER_DEPLOY_ENABLED=true`.

**VPS architecture:**
OpenClaw and Clawvis Hub on the same VPS. Communication via `http://host.docker.internal:3333` (Docker resolves to host automatically). Single exposed port: Hub on 8088 (or 80 with nginx reverse proxy).

## Phase 5 — Skills ecosystem

**Exit criterion:** public Clawvis skills are published and discoverable outside the repo.

> Post-launch phase. Prerequisite: Phase 4 complete, at least 5 stable public skills documented.

### 5A — Quality & marketplace criteria

A skill is "marketplace-ready" if:
- It has a `README.md` with description, usage examples, prerequisites
- It is tested via `skill-tester` (≥ 80% coverage of nominal cases)
- It has no instance dependencies (no secrets, no hardcoded paths)
- It has a `CHANGELOG.md` or changelog section in README
- Its name follows `<domain>-<action>` (e.g. `kanban-implementer`, `brain-pulse`)

### New skill creation workflow

```
1. Create folder in clawvis/skills/<name>/ or instances/<name>/skills/<name>/
2. clawvis skills sync   ← auto-symlink to ~/.openclaw/skills/
   (or automatic via git hook post-merge)
3. Test via skill-tester
4. If public + stable → add skill.json metadata for marketplace
```

## Production hardening — deployment checklist

> Reference: `docs/adr/0004-production-deployment-pitfalls.md` (8 documented pitfalls)

Verify these on every new deploy or upgrade:

### Before first instance deploy

- [ ] **Symlink**: if `instances/<name>` is a symlink, add explicit volume overrides in `docker-compose.override.yml` for each service that reads that path
- [ ] **`hub_settings.json`**: update `projects_root` with real projects path on host (default = invalid placeholder)
- [ ] **HOME env**: any container using `Path.home()` must have `HOME=<user_home>` in environment + volume mount for required home paths

### Before nginx reload

- [ ] Always `export` variables before `envsubst` (non-exported = empty in child process)
- [ ] Scope `envsubst` with explicit list: `envsubst '${HUB_ROOT} ${LAB} ${OPENCLAW_GATEWAY_TOKEN}'`
- [ ] Run `nginx -t -c <conf>` — pid warning is a false positive; other errors are real
- [ ] After `kill -HUP`, check `nginx -error.log` to confirm reload

### After any source change

- [ ] Rebuild explicitly: `docker compose build <service>`
- [ ] Force recreate: `docker compose up -d --force-recreate <service>`
- [ ] Verify deployed JS contains expected strings (e.g. `docker exec hub grep -r "expected-string" /usr/share/nginx/html/`)

### Nginx — required location blocks for Hub SPA

- [ ] `location /assets/` — BEFORE `/hub/` (Vite absolute asset paths)
- [ ] Regex for root statics: `location ~ ^/(clawvis-mascot\.svg|favicon\.ico|favicon\.png)$`
- [ ] Redirects for SPA sub-paths accessed directly: `location = /chat/ { return 301 /hub/#/chat; }`
- [ ] Check for duplicates BEFORE adding a location block — nginx rejects duplicates at startup
